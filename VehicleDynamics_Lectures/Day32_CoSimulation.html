왜 이 수식이 필요한가?

💡 수식 유도 읽는 법 — 각 STEP의 파란색 라벨은 "왜 이 변환을 하는가"를 설명합니다. 수식만 보지 말고 동기 → 변환 → 의미 순서로 읽으세요. 눈으로 3번 읽는 것보다 손으로 1번 쓰는 것이 기억에 4배 효과적입니다.

자동차의 물리적 거동(서스펜션 운동, 차체 진동)은 Simpack이 잘 계산합니다. 하지만 반능동 댐퍼(CDC)나 에어 서스펜션처럼 제어 로직이 필요한 시스템은 Simulink가 전문입니다. Co-Simulation은 두 소프트웨어를 매 통신 스텝마다 연결하여, 물리 + 제어를 동시에 시뮬레이션하는 기법입니다.

Step 1: Co-Simulation 아키텍처
Simpack(Plant)←→Simulink(Controller)매Δtcomm마다:Simpack→센서값→Simulink→제어명령→SimpackSimpack(Plant) ←\rightarrow Simulink(Controller) \text{매} \Deltat_comm \text{마다}: Simpack \rightarrow \text{센서값} \rightarrow Simulink \rightarrow \text{제어명령} \rightarrow SimpackSimpack(Plant)←→Simulink(Controller)매Δtc​omm마다:Simpack→센서값→Simulink→제어명령→Simpack

이 항의 의미: 통신 주기 Δt_comm마다 데이터를 교환합니다. Simpack은 현재 상태(변위, 속도, 가속도)를 Simulink에 전달하고, Simulink는 제어 명령(댐퍼력, 에어스프링 압력)을 Simpack에 전달합니다.

부호/변환 이유: Δt_comm은 두 소프트웨어의 내부 적분 스텝보다 크거나 같아야 합니다. 너무 크면 안정성 문제, 너무 작으면 계산 시간 증가.

이전 단계와의 연결: Day 21 Skyhook 제어에서 '이상적 제어력'을 수식으로 유도했는데, 이것이 Simulink 제어 블록으로 구현됩니다.

부호/변환 이유: FMI 2.0 표준: 소프트웨어 독립적 인터페이스. Simpack FMU 내보내기 → Simulink에서 FMU Import. 가장 범용적 방법.

이 항의 의미: Simpack이 플랜트의 상태를 출력하면, Simulink가 제어 법칙에 따라 액추에이터 명령을 계산하고, 이 명령이 다시 Simpack에 적용됩니다. 폐루프(Closed-loop) 시뮬레이션이 완성됩니다.

부호/변환 이유: 인과성(Causality): 센서→제어기→액추에이터→플랜트→센서의 순환. 시간 지연(1 step delay) 고려 필요.

Step 4: Co-Simulation 실행 & 검증
PassiveDamper결과vsCDCCo−Sim결과 비교 평가지표:RMSz¨s(승차감),DynamicLoadVariation(안전성)Passive Damper \text{결과} vs CDC Co-Sim \text{결과 비교 평가지표}: RMS z̈_s (\text{승차감}), Dynamic Load Variation (\text{안전성})PassiveDamper결과vsCDCCo−Sim결과 비교 평가지표:RMSz¨s​(승차감),DynamicLoadVariation(안전성)

이 항의 의미: 제어 없는(Passive) 결과와 제어 있는(Co-Sim) 결과를 비교하여 제어 효과를 정량 평가합니다. 승차감(RMS 가속도 감소율)과 안전성(동적 하중 변동 감소율)이 핵심 지표입니다.

부호/변환 이유: 좋은 제어기 = RMS z̈_s 감소 + Dynamic Load Variation 미증가. 트레이드오프 개선이 최종 목표.

이전 단계와의 연결: Day 21의 승차감 vs 조종안정성 트레이드오프가 Co-Sim으로 정량 평가됩니다.

📐 상세 유도과정 (Step-by-Step)

중간 과정 완전 전개

① Co-Simulation 인터페이스

$$\text{SIMPACK} \xleftrightarrow[\Delta t_{comm}]{\text{FMI/FMU}} \text{MATLAB/Simulink}$$
$$\text{통신 주기: } \Delta t_{comm} = n \times \Delta t_{solver}$$

FMI 2.0 표준으로 이기종 솔버 간 데이터 교환. Co-sim 안정성: ΔT_comm ≤ 1/10 × 최소 고유주기

② 입출력 변수 매핑

$$\text{SIMPACK }\to\text{ Simulink: } F_{susp}, z_{body}, \dot{z}_{body}, a_{body}$$
$$\text{Simulink }\to\text{ SIMPACK: } F_{actuator}, T_{motor}, P_{brake}$$

기계 시스템(SIMPACK)과 제어 시스템(Simulink) 분리. 각자 최적 솔버 사용 가능

③ 통신 지연 보상

$$\hat{y}(t+\Delta t) = y(t) + \dot{y}(t) \cdot \Delta t \quad (\text{외삽})$$

영차 유지(ZOH) vs 1차 외삽. 고주파 현상에서는 외삽 필수. 불안정 시 ΔT_comm 축소

④ 실시간 요구사항 (HiL)

$$\text{실시간 비율: } RT_{ratio} = \frac{t_{wall}}{t_{sim}} \leq 1.0$$
$$\text{Step size: } \Delta t \leq 1\text{ms (서스펜션)}, \leq 0.1\text{ms (타이어)}$$

HiL 테스트: ECU 실물 + 가상 차량 모델. SIMPACK RT 솔버로 실시간 달성

FMI 교환 변수: $y(t_k) = f(u(t_{k-1}))$ → 이전 스텝 입력으로 현재 출력 계산

통신 스텝: $\Delta t_{comm} \leq \frac{1}{10 f_{max}}$ → 최고 관심 주파수의 10배 이상 샘플링

■ FMI (Functional Mock-up Interface)

Co-Simulation: 각 FMU가 자체 솔버 보유 → $\Delta t_{comm}$ 간격으로 변수 교환

Model Exchange: 하나의 마스터 솔버가 전체 적분 → 더 정확하지만 설정 복잡

르노코리아 실무 Tip: Simpack-MATLAB Co-Simulation 시 통신 스텝을 너무 크게 하면 에너지 발산이 발생합니다. $\Delta t_{comm} = 0.001$ s부터 시작하세요.

시뮬레이션 도구 간 표준 인터페이스 (Modelica Association 주관)

FMU = FMI 규격에 맞춘 실행 가능 모듈 (.fmu 파일)

■ Co-Simulation vs Model Exchange

Co-Simulation: 각 FMU가 자체 솔버 보유 → 데이터만 교환

Model Exchange: 마스터 솔버가 모든 FMU를 통합 계산

■ Simpack ↔ Simulink 연동

Simpack → FMU 2.0 Co-Sim 내보내기

Simulink에서 FMU Import Block으로 로드

통신 주기 Δt: 일반적으로 1ms (1kHz) 이하 권장
